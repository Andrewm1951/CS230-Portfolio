# CS 230 Portfolio Reflection – The Gaming Room

The Gaming Room is a company that wanted to expand their game Draw It or Lose It beyond Android to multiple platforms. The game is a multiplayer application where teams guess images that are revealed over time. Because of this, the software needed to support multiple users at once, run efficiently across different devices, and handle a large amount of image data while maintaining performance.

One thing I think I did particularly well in this project was clearly organizing the design and breaking down each part of the system. I was able to explain how the client-server model works, how different platforms compare, and what technologies would best support the application. I also made sure to keep the explanations simple and direct, which makes the document easier to understand.

Working through the design document helped me better understand how software is planned before it is built. Instead of jumping straight into coding, it forced me to think about structure, performance, and scalability first. This made it easier to understand how different parts of the system connect and what decisions need to be made ahead of time.

If I could go back and revise one part of my work, I would improve the platform evaluation section. While I covered the main points, I could have gone into more detail about real-world use cases and provided stronger comparisons between each operating system. This would make the recommendations more detailed and supported.

When interpreting the user’s needs, I focused on what the client wanted most: cross-platform support, smooth performance, and the ability to handle multiple players at once. I used those requirements to guide my decisions, such as recommending a Linux server and a web-based client system. Considering the user’s needs is important because it ensures the final product is useful and meets expectations instead of just being technically correct.

When designing the software, I approached it step by step by breaking the system into smaller parts, such as server, client, storage, and memory management. This made the process easier to understand and allowed me to focus on one piece at a time. In the future, I would continue using this approach along with researching best practices and comparing different technologies before making decisions.

Overall, this project helped me understand how to design software systems more effectively and how to communicate those designs clearly. It also showed me the importance of planning before development and how different components work together to create a complete application.
